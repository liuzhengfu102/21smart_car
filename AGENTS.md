## 4. 标准开发流程

### 4.1 开始任务

1. 阅读本文件和任务相关源码。
2. 执行 `git status --short --branch`，确认工作区状态。
3. 判断任务是否会修改板卡主版本文件。
4. 若会修改 `car.cpp` 或其他以板卡为主版本的文件，先进行本地与板卡同步检查。
5. 查看相关提交历史，理解现有设计意图后再修改。

### 4.2 主版本同步规则

板卡端 `car.cpp` 当前视为运行主版本。本地仓库是可版本化的开发镜像。

修改 `car.cpp` 前，必须比较本地与板卡文件哈希：

```powershell
# Windows 本地
Get-FileHash .\car.cpp -Algorithm MD5

# Orange Pi 板卡
ssh root@192.168.3.57 "md5sum /home/orangepi/Desktop/smartcar/car.cpp"
```

处理原则：

- 哈希一致：可以继续本地修改。
- 哈希不一致：先停止修改，读取两端差异来源。
- 板卡版本确认更新：先将板卡文件拉到临时文件并比较，禁止直接覆盖本地未提交内容。
- 本地存在未提交修改：必须先保存差异或提交，禁止直接执行覆盖式 `scp`。

安全拉取示例：

```powershell
scp root@192.168.3.57:/home/orangepi/Desktop/smartcar/car.cpp .\car.cpp.board
git diff --no-index -- .\car.cpp .\car.cpp.board
```

确认板卡版本应成为本地基线后，再由刘征孚同意合并或替换。

### 4.3 实现与自查

代码修改后至少检查：

- 控制逻辑是否符合任务目标，默认行为是否发生意外变化。
- 数组、缓冲区、共享内存偏移和对象数量是否有边界检查。
- 变量是否初始化，状态是否能在无目标、数据中断和异常输入后正确复位。
- 文件描述符、映射内存和串口资源是否在所有路径正确释放。
- 整数转换、符号位、溢出、除零和浮点转整数是否安全。
- 串口是否可能出现短写、发送失败或越界控制值。
- 信号处理函数是否调用了非异步信号安全操作。
- 高频循环是否存在无休眠空转、过量输出或无意义系统调用。
- 修改是否兼容 ARM64、GCC 11.4.0、CMake 3.22.1 和 OpenCV 4.5.4。

### 4.4 验证层级

按任务风险选择验证，能执行的项目不得省略：

1. 静态检查：阅读差异、搜索调用点、检查边界与状态转换。
2. 本地文档检查：`git diff --check`，确认 Markdown 结构和命令无明显错误。
3. 本地语法或脚本检查：例如 `python -m py_compile shm_writer.py`。
4. 板卡编译：需刘征孚同意后，在 Orange Pi 上执行 CMake 和 Make。
5. 板卡运行：需再次确认硬件、串口、摄像头和车辆安全条件后执行。

本地 x86 环境生成的二进制不能替代 ARM64 板卡编译结果。

### 4.5 收尾

1. 执行 `git diff --check`。
2. 审阅完整差异，确保未包含敏感信息和无关文件。
3. 创建 Git 提交。
4. 尝试推送 GitHub。
5. 汇报提交哈希、推送结果、验证结果和未完成事项。

## 8. 共享内存协议

### 8.1 基本信息

| 项目 | 值 |
|---|---|
| POSIX 名称 | `/shm_road_coords` |
| Python `SharedMemory` 名称 | `shm_road_coords` |
| 固定映射大小 | 4096 字节 |
| 当前字节序 | 依赖生产端与消费端本机原生小端格式 |

### 8.2 二进制布局

```text
uint32_t road_num
repeat road_num times:
    float x
    float y
uint32_t obj_num
repeat obj_num times:
    uint32_t c
    float x1
    float y1
    float x2
    float y2
```

| 区域 | 单项大小 | 说明 |
|---|---:|---|
| `road_num` | 4 字节 | 原始道路点数量 |
| 道路点 | 8 字节 | 两个 `float`：`x, y` |
| `obj_num` | 4 字节 | 目标数量，位置取决于 `road_num` |
| 检测目标 | 20 字节 | `uint32_t c` 加四个 `float` 坐标 |

目标类别：

| 类别值 | 含义 |
|---:|---|
| `0` | 金币 |
| `1` | 车 |
| `2` | 行人 |

协议修改要求：

- 生产端和消费端必须同步修改，不得只改一侧。
- 修改字段、字节序、对齐或容量时，必须补充协议版本或兼容方案。
- 读取前必须验证数量与剩余缓冲区容量，不能信任外部写入的 `road_num` 和 `obj_num`。
- 生产端可能正在写入时，当前协议没有序号、锁或双缓冲保证一致帧；涉及稳定性优化时优先评估该问题。

- 道路点只保留 `y` 位于 `[350, 480]` 的点。
- 目标当前只保留包围框纵向范围与 `y=480` 相交的对象。

## 10. 当前控制逻辑基线

以下描述对应当前 `car.cpp`，代码变化后必须同步更新：

1. 道路中心：对有效道路点拟合 `x = a*y + b`，用近端中线和远端趋势计算预瞄目标，结果限幅后保存为 `base_road_x`。
2. 目标优先级：面积超过阈值的车或行人优先于金币。
3. 同类选择：在通过面积筛选的候选中，优先选择包围框下边缘 `y` 最大的目标。
4. 障碍规避：目标中心在图像左侧时增大中线目标向右规避，在右侧时减小中线目标向左规避；避障偏移根据障碍接近程度和居中程度在 `180` 到 `280` 间动态计算。
5. 金币追踪：根据金币中心与图像中心的偏差修正道路中线，并使用 EMA 平滑。
6. 状态保持：障碍消失后继续保持最后避障结果 `0.6` 秒。
7. 最终输出：所有结果再经过一次 EMA 平滑后发送。
8. 信号退出：收到 `SIGINT` 时尝试发送停车帧并关闭串口。

4. 在板卡重新配置并编译。
5. 检查编译产物架构和时间戳。
6. 在确保车辆架空、急停可用、串口目标正确后运行。
7. 记录结果；失败时恢复备份。

不得未经确认直接执行“传输 + 编译 + 运行”的组合命令。

## 12. 测试与调试

### 12.1 共享内存模拟

`shm_writer.py` 用于创建或连接 `shm_road_coords`，循环写入模拟道路点和目标。使用前注意：

- 当前模拟道路点 `y` 为 `300, 315, 330, 345, 360`，按 `car.cpp` 的道路筛选规则通常仅 `y=360` 会被保留。
- 当前模拟目标框坐标全部为 `0`，不会通过正式目标筛选。
- 脚本退出时会调用 `unlink()` 删除共享内存；不得与正式推理进程同时运行。
- 脚本使用原生格式打包，默认假设生产端与消费端架构和字节序一致。

### 12.2 虚拟串口

虚拟串口路径可能由 `socat` 等工具动态生成。测试时必须：

- 先确认当前 `/dev/pts/*` 两端映射。
- 不得把历史路径 `/dev/pts/8` 当成固定设备。
- 捕获原始 8 字节帧，核对帧头、字段顺序和帧尾。
- 测试停车帧与 `SIGINT` 退出行为。

### 12.3 硬件运行安全

- 首次运行新控制逻辑前应将驱动轮架空或断开动力。
- 明确急停方式，并确保操作人员可立即触发。
- 确认串口设备不是系统控制台或其他外设。
- 摄像头、NPU 模型和共享内存生产端就绪后，再启动车辆控制程序。
- 当前曾记录 `lsusb` 未发现摄像头；每次硬件测试前应重新检查，不能视为永久状态。

## 13. 已知风险与审查重点

修改相关代码时优先检查以下风险：

- 共享内存没有帧版本、长度、序号、锁或校验，可能读取到生产端写入中的不一致数据。
- 对共享内存直接使用 `reinterpret_cast` 读取，需关注对齐、严格别名和协议兼容性。
- 串口 `write()` 当前只判断负值，未处理少于 8 字节的短写。
- 主循环当前没有显式休眠，数据缺失时仍可能高频轮询和发送。
- 转向值在拆分成两个字节前没有明确限幅，必须与底层协议确认合法范围。
- `SIGINT` 处理函数当前调用串口、输出流和 `exit()`，这些操作并非都具备异步信号安全性。
- 无有效道路点时会沿用上一次 `base_road_x`，这是当前行为，修改前需确认失联安全策略。
- 串口打开失败后程序仍继续运行，只是不发送数据；是否应退出待产品策略确认。
- `shm_writer.py` 的模拟数据与当前筛选条件不完全匹配，不能直接证明目标控制逻辑有效。

这些条目是审查清单，不代表可以脱离任务范围擅自重构；发现高风险问题时先说明影响，再做最小范围修复。

## 14. 编码规范与兼容性

- C++ 修改应保持当前工具链可编译，不使用 GCC 11.4.0 不支持的语言特性。
- 优先使用 RAII 管理文件描述符、映射和其他资源。
- 协议字段优先使用固定宽度整数类型，如 `uint32_t`、`uint8_t`。
- 新增魔法数字必须提取为有语义的常量。
- 高频路径避免不必要的动态分配、字符串拼接和终端输出。
- 不引入大型依赖，除非现有标准库或项目依赖无法合理解决问题。
- 注释重点解释约束、原因和协议，不重复代码表面含义。
- 保持源码和文档 UTF-8 编码。

### 14.1 OpenCV 命名空间注意事项

`getTickCount()` 和 `getTickFrequency()` 属于 `cv` 命名空间：

```cpp
cv::getTickCount();
cv::getTickFrequency();
```

若移除 `using namespace cv;`，必须保留上述 `cv::` 前缀，否则板卡端 GCC/OpenCV 环境会编译失败。
